# 写给计算机科学家的会计学

作者：Martin Kleppmann，发表于 2011 年 3 月 7 日

---

每个受过教育的人都应该对会计学有基本的了解。就像数学、科学、编程、音乐、文学、历史等一样，它是帮助你理解世界的那些东西之一。虽然跟钱打交道并不怎么有趣，但它是生活中不可避免的一部分，所以你不妨花几分钟了解一下。

遗憾的是，在我看来，大多数会计师的讲解方式都非常糟糕，让人难以理解；这个领域充满了行话、缩写词和奇怪的历史遗留问题。就连《会计傻瓜书》也让我头晕目眩。这些东西真的有那么难吗？（我们搞计算机的人可能在糟糕的解释和行话方面也有同样的罪过。问题是，一旦你对某个领域非常熟悉，就很难想象在你理解它之前是怎么看待这些事情的。）

最终我搞明白了：**基础会计学就是图论**。传统的财务信息表示方式极好地掩盖了这种结构，但一旦我发现它不过是一个图，一切突然就变得清晰了。

我是一个计算机科学家，我一直用图来思考问题。如果当初有人用这种方式给我解释就好了！这能省去我好多困惑。所以我想尝试弥补这一点。如果你喜欢图，那么当你读完这篇文章时，你应该已经掌握了理解一家小公司/初创企业的财务报表所需的全部知识（甚至可以用你自己选择的电子表格或编程语言自行计算）。

这其实没那么难。让我们开始吧！

---

## 账户 = 节点，交易 = 边

假设你去百吉饼店，用公司信用卡买了一个超级俱乐部百吉饼，花了 5 美元。你还去了一家硅谷的随机初创公司，从他们那里二手买了一把多余的 Aeron 椅子，花了 500 美元（从公司账户开了一张支票）。这是两笔交易。

每笔交易都是我们图中的一条**边**，边上标有金额。

一条边总是从一个节点指向另一个节点。这些节点是什么呢？嗯，你可以按自己的喜好来定义它们（虽然有一些惯例）。现在，我们这样说：

![图1：百吉饼和椅子的初始交易图](https://martin.kleppmann.com/2011/03/accounting1.dot.png)

让我们补充一些细节。你用公司账户支付了 5 美元的信用卡账单。而公司账户里的钱最初是从哪里来的呢？啊，我明白了，你投入了 5,000 美元的个人储蓄来创办公司。好了，现在图看起来是这样的：

![图2：加入初始投资和信用卡还款后的图](https://martin.kleppmann.com/2011/03/accounting2.dot.png)

希望到目前为止都很容易理解。钱沿着箭头的方向流动。

你又饿了，去墨西哥卷饼店买了一个超级卷饼，用信用卡花了 8 美元。现在我们可以为墨西哥卷饼店创建另一个节点，但这开始变得有点混乱——我们并不真的关心花在百吉饼上和花在卷饼上的钱各是多少。让我们把它们合并为"食物"。另外，"随机初创公司"这个名称也有点没用——我已经忘了那 500 美元是干什么用的了。让我们改叫它"家具"。

![图3：将消费归类为食物和家具后的图](https://martin.kleppmann.com/2011/03/accounting3.dot.png)

看，这完全没问题。我们可以有代表实际银行账户或卡的节点，有代表人或公司的节点，还有代表"食物"或"家具"这类抽象类别的节点。把它们全部扔进同一个图里就好。

还要注意，同一对节点之间可以有多条边。你可以单独追踪每条边，也可以简单地把它们加起来。（用信用卡在食物上总共花了 13 美元。）

---

## 账户有余额

这个图中的每个节点在会计术语中都是一个**账户**（无论它是否由银行持有），每个账户都有一个**余额**。余额是每个账户的单个数字，它完全由进出该账户的交易决定：

- 在时间的起点，每个节点的值都是零。
- 在每个节点上，对于每条**流入**的边，将该边的标签值加到节点的值上；对于每条**流出**的边，从该值中减去该边的标签值。
- 处理完所有边之后，每个节点上的值就是该账户的余额。

我们的图现在看起来是这样的：

![图4：显示各账户余额的图](https://martin.kleppmann.com/2011/03/accounting4.dot.png)

注意账户余额有两个很好的性质：

- 因为每笔交易都出现两次——一次为正，一次为负——所以**所有账户余额的总和始终为零**。
- 如果你将节点集合划分为任意两个不相交的子集，然后将每个子集中所有余额相加，那么一个子集的总和永远是另一个子集总和的相反数（因为，毕竟它们加起来必须等于零）。

这些性质对于检查你的数字是否正确很有用；如果它们被违反了，那就是"你做错了"。（这就是会计师所说的"平衡账目"。）

## 做生意

吃完百吉饼和卷饼后，你出去跟一些潜在客户谈。嘿，他们喜欢你的产品！产品标价 5,000 美元，你把它卖给了两个大企业客户。一个客户立刻付了钱（太棒了！）；另一个客户先预付了 2,500 美元，但坚持要求在他们付清余款之前，你必须实现你愚蠢地承诺过的那个额外功能。

所以你从客户那里收到了 5,000 + 2,500 美元的现金，直接汇入了公司银行账户。让我们把它加到图上：

![图5：加入客户付款后的图](https://martin.kleppmann.com/2011/03/accounting5.dot.png)

但这不太对。每个客户的价格都是 5,000 美元，但现在看起来你收了两个不同的价格。我们该如何表示跟客户 2 的约定呢？

解决方案是将这笔交易**分解为两笔独立的交易**：**销售**（买方同意购买，但没有实际资金转手）和**付款**（现金实际进入你的银行账户）。我们可以这样画：

![图6：将销售和付款分离后的图](https://martin.kleppmann.com/2011/03/accounting6.dot.png)

看到我做了什么吗？我刚刚发明了一个新节点，笼统地叫它"销售"，然后把实际 5,000 美元的销售作为从"销售"账户到客户账户的交易加了上去。添加这个额外节点并没有改变你的银行余额。

当你从余额的直观含义来思考时，这是有道理的。

每个客户账户的余额是他们欠你的金额：客户 1 已经完全付清了（他们的流入和流出交易加起来相等），所以余额为零；客户 2 已经合同约定给你 5,000 美元，但目前只给了一半，所以余额是 2,500 美元。

而销售账户的余额是你卖出的东西的价值。或者说，负值。这看起来有点奇怪……但我稍后再说这件事。（顺便说一句，如果你想分别追踪不同客户或不同产品的销售，没问题——只需添加对你有意义的节点即可。只要确保每笔交易只出现一次作为一条边，否则你就是在捏造数据！）

## 完成示例

为了圆满收尾，让我给这个故事再补充一些事件（= 给图再加几条边）。

你不仅做成了几笔销售，现在还收到了 Y Combinator 的 20,000 美元投资——恭喜！你和你的联合创始人现在可以给自己发工资了。你从公司账户取出了 8,000 美元。

然后你找了一位公司会计师，他们跟你讲了一大堆行话。出于某种奇怪的原因，他们痴迷于正确核算你的办公椅；他们希望它在四年内折旧，也就是说它的价值在这段时间内逐渐减到零。好吧，你说（尽管你根本不在乎你的椅子四年后值多少钱——到那时你肯定已经成为下一个谷歌或 Facebook 了，你会有其他比椅子更需要担心的事情）。

最终的图现在看起来是这样的：

![图7：完整示例图，包含 YC 投资、工资和折旧](https://martin.kleppmann.com/2011/03/accounting7.dot.png)

注意我是如何表示这些交易的：

- 我把你的创始人投资和 Y Combinator 的投资合并在一起，放在"资本"标题下。简单来说，这是你通过出售公司股份而不是向客户出售产品或服务而获得的钱。像往常一样，如果你愿意，你可以把创始人和 YC 分成独立的账户。
- 我把工资单（薪水）表示为直接从银行账户出去的钱。实际上由于税收、医保、福利等原因会复杂一些，但原理是一样的。只是在图里多了一些节点和边。
- 我让一年的折旧（500 美元的四分之一 = 125 美元）从"家具"账户中离开。直观地说，这意味着"家具"账户的余额是你的家具现在仍然具有的价值。每年，你再添加一条 125 美元的边从"家具"指向"折旧"，直到四年后，"家具"的余额降到零（假设在这期间你没有为了追求世界统治而再买更多椅子）。

## 利润表

如果你已经感到疲倦了，我不会怪你。但好消息是：**我们已经完成了图的构建**！现在我将向你展示这种图表示如何映射到管理公司最常用的两份标准财务报表：**利润表**（"P&L"）和**资产负债表**。这很有用，因为作为初创公司创始人，你迟早要跟你的投资人/顾问讨论这些文件，所以你不妨搞清楚它们到底是什么意思。

为了生成这些报表，我需要拿出蜡笔。这是跟之前一样的图，不过节点被涂上了颜色：

![图8：按颜色分类的完整图](https://martin.kleppmann.com/2011/03/accounting8.dot.png)

解释颜色（括号里加上会计术语，因为你很可能会遇到这些词）：

- **绿色**：代表**你拥有的东西**（"资产"），例如银行里的钱，或者你买来的可以再卖掉的东西，比如家具。绿色还代表欠你钱的人/公司（"应收账款"/"债务人"，比如客户 2），以及你欠钱的人/公司（"负债"/"债权人"，比如你即将到来的信用卡账单——那个卷饼的钱）。
- **蓝色**：代表**你的产品/服务销售**（"收入"）和**你花掉的钱**且不会回来的（"费用"/"开销"）。办公椅是绿色的，因为如果你想的话可以再卖掉它，但百吉饼是蓝色的，因为一旦你买了（并吃掉）它，就完了——无法回头。
- **粉色**：代表来自投资人（或你自己）的钱，是你通过出售股份获得的（"资本"）。（如果你获得银行贷款，那是绿色的，不是粉色的，因为你欠银行要还的钱。）

你的每个节点都应该恰好属于这些类别之一。如果不是，那要么是哪里出错了，要么是你发现了一些我还不知道的会计领域。

有了这些颜色设定，**利润表就是所有蓝色节点的列表**，而公司的利润或亏损就是所有蓝色节点余额的总和。按照我们的计算方式，负值代表利润，正值代表亏损。这有点让人困惑，所以你在报告数字时通常会翻转符号（这样利润就是正数）。

用标准方式写出来，我们的利润表看起来是这样的：

| 收入 | |
|:---|---:|
| 销售 | $10,000 |
| **总收入** | **$10,000** |
| | |
| **费用** | |
| 工资 | $8,000 |
| 折旧 | $125 |
| 食物 | $13 |
| **总费用** | **$8,138** |
| | |
| **利润/亏损** | **$1,862** |
| （= 总收入 - 总费用） | |

含义相当直观。你卖出了价值 10,000 美元的东西，而在这个过程中只花了 8,138 美元，所以你赚了 1,862 美元的利润。

利润表是**在一段时间内**计算的（通常是一个月、一个季度或一年），比较两个不同时期通常很有趣。要计算某个时期的利润表，将你的交易过滤为只包含该时期内发生的交易，然后只累加那些交易的账户余额。

**有一点需要注意**：利润跟你的银行账户**没有任何关系**。银行账户是一个绿色节点，但我们这里只看蓝色节点。在这个例子中，你的银行里有 23,995 美元，尽管投资人投入了 25,000 美元：你实现了利润，但银行里的钱仍然比之前少了，因为客户 2 还没有全额付款。这就是为什么一家公司可能**有利润但仍然会缺钱**！

## 资产负债表

资产负债表不如利润表直观，但它是一份相当有力的文件。它总结了公司目前拥有什么和不拥有什么，以及原因。

还记得我之前说过，将节点划分为两个不相交的集合，它们的余额总和加起来为零吗？这正是资产负债表上发生的事情。我们取图中的所有节点；在一边我们考虑所有绿色节点，在另一边考虑所有蓝色和粉色节点。所有蓝色和粉色节点余额的总和等于所有绿色节点余额总和的相反数。

现在，按照惯例，会计师会**翻转所有蓝色和粉色节点余额的符号**，这意味着两边的总和最终相等。这就是它被称为**资产负债表**的原因。

在我们的例子中，它看起来是这样的：

| 资产 | |
|:---|---:|
| 银行账户 | $23,995 |
| 应收账款 | $2,500 |
| 家具 | $375 |
| **总资产** | **$26,870** |
| | |
| **负债** | |
| 信用卡 | $8 |
| **总负债** | **$8** |
| **总资产减总负债** | **$26,862** |
| | |
| **所有者权益** | |
| 利润/亏损 | $1,862 |
| 资本 | $25,000 |
| **总权益** | **$26,862** |

上半部分（资产和负债）对应图中的绿色节点，而下半部分包含粉色节点（资本）和所有蓝色节点的总和。我们已经在上面的利润表中展示了蓝色节点的所有细节；在资产负债表上，我们可以把它们汇总成一个数字。

这里又发生了一些符号翻转：我把负债、权益和利润/亏损都写成了翻转符号后的形式（这通常但不总是会使数字变为正数）。这并没有改变图结构的任何本质，只是把它们放进了传统的格式中。

那么如何解读资产负债表呢？你可以从中读出很多信息。你可以看到银行里有多少钱，以及这些钱中有多少已经承诺给了别人（负债）。你可以看到银行里的钱有多少来自投资人，有多少来自销售。它还显示了由于已成交但尚未全额付款的销售，即将有多少钱会进来。

**资产负债表的总数是你公司价值的下限**。这是一个非常悲观的数字——它假设你的团队、你的技术、你的品牌等等都恰好值零；如果你的公司从投资人那里融资，你的估值会比资产负债表上的数字高得多，因为那个估值以胡乱猜测的形式包含了团队、技术、品牌等的价值。在成熟的公司里，你可以在资产负债表上找到"无形资产"，但由于它们很难估值，我觉得除非你知道自己在做什么，否则没必要去费这个心。

## 一、学术理论基础

### 1. David Ellerman：最系统的数学形式化

这是最核心的学术论文。Ellerman 在 **1985 年**首次发表，后于 **2014 年在 arXiv 上重新整理**为《On Double-Entry Bookkeeping: The Mathematical Treatment》。

**核心发现：**

- 复式记账法（DEB）隐式使用了一个精确的数学结构——**差值群**（Group of Differences），即有序对 `[a // b]`（也就是 T 账户的左/右两侧）
- 他将其命名为 **Pacioli 群**（以纪念 1494 年发表复式记账系统的 Luca Pacioli）
- 该结构直到 **19 世纪**才被数学家（William Rowan Hamilton）抽象形式化，而商人已经用了五百年

**最重要的推广：**

- Ellerman 证明了 DEB 可以**直接推广到多维向量**，即不需要共同的价值度量单位（如货币），就能对不同物理量的库存（如 40 个成品、50 个半成品、$9000 现金）进行复式记账
- 这反驳了"复式记账必须以货币为统一计量单位"的传统观点

### 2. Rambaud 等人：《Algebraic Models for Accounting Systems》(2010)

世界科学出版社出版的专著，使用**抽象代数**工具为复式记账构建严格模型：

- **有向图**（digraphs）
- **自动机**（automata）
- **幺半群**（monoids）
- **商结构**（quotient structures）

这是目前该领域最全面的数学处理之一。

### 3. 四重记账与国民经济核算

**OpenRiskManagement** 将图表示从单个实体的**双重记账**扩展到**四重记账**（Quadruple-Entry Accounting），用于多个经济实体之间的交易网络建模。这直接应用于：

- 国家统计账户编制（GDP 计算等）
- 社会会计矩阵（Social Accounting Matrix）

使用**关联矩阵**（Incidence Matrix）表示账户与交易之间的关系，列和为零即对应"借贷平衡"约束。

## 二、知识图谱与审计应用

### 会计知识图谱（2022）

一篇 SCItePress 论文《Towards a Knowledge Graph-specific Definition of Digital Transformation》描述了将传统 ERP 中的会计交易迁移到 **RDF 知识图谱**的实践：

- 将账户作为图中的节点，交易作为带权重的边

### 1. 会计理论维度：只有 PTA 三剑客真正实现了 Kleppmann 的图模型

## 二、与高性能关系型数据库的直接对比

### 核心差异：计算模型不同

| 维度 | PTA (纯文本) | 关系型数据库 (Firefly III/Akaunting) |
|------|------------|-----------------------------------|
| **存储** | 纯文本文件 | MySQL/PostgreSQL 表 |
| **查询方式** | 每次运行**全量解析**文本文件 → 构建内存结构 → 计算 | **索引查询**，只读取相关行 |
| **计算位置** | 应用端（客户端） | 数据库引擎端 |
| **并发** | 无（单用户文件锁） | 强（MVCC 事务隔离） |
| **增量更新** | ❌ 不支持（每次都从头来） | ✅ 天然支持 |

### 那实际差异有多大？

以 **Firefly III**（MySQL/PostgreSQL 后端，23.6k⭐）为例：

- **日常查询**（如查看本月支出）：数据库索引命中，**毫秒级**
- **Firefly III 启动/页面加载**：需要初始化 Laravel 应用、查询多表关联、渲染 Blade 模板，实际用户感知的延迟可能在 **200-500ms**

对比 PTA：

- **Ledger 查询 12 万交易余额**：**120ms**（比数据库索引查询慢一个数量级，但仍是人类无感知的）
- **hledger 同样查询**：**180ms**

**关键点**：在个人财务的数据规模下（通常 < 50k 交易/年），**两者都快得绰绰有余**。你不会感知到 PTA 的"性能劣势"——因为你的账本每年可能只有几千条交易，任何工具都能在 1 秒内完成。

## 三、PTA 的真正性能瓶颈在哪里？

### 不是"文本解析慢"，而是**没有增量机制**

```
数据库会计软件的工作流程：
  启动 → 保持连接 → 查询1(10ms) → 查询2(10ms) → 查询3(10ms)

PTA 的工作流程：
  启动 → 读取整个文本文件 → 解析 → 构建内存结构 → 查询1(完成)
       → 读取整个文本文件 → 解析 → 构建内存结构 → 查询2(从头再来)
       → 读取整个文本文件 → 解析 → 构建内存结构 → 查询3(从头再来)
```

PTA 每次命令执行都是**独立的、无状态的进程**，必须从头解析整个文件。这是设计上的选择（Unix 哲学：每个命令做一件事，做完即走），而不是文本格式本身的限制。

### 实际影响

一位 Beancount 用户的真实反馈：
> "3 年后文件增长，我不得不每年'关账'（close books）以保持查询速度。"

另一位 hledger 用户的测试：
> "我的脚本需要执行 hledger 多次，每次 4 秒，累积起来变得很慢。同样数据用 Ledger 只需 381 毫秒。"

## 四、PTA 社区的应对方案

面对性能挑战，PTA 社区没有"换数据库"，而是从多个方向优化：

### 1. Beancount v3：C++ 核心重写（即将发布）

| 版本 | 核心语言 | 预期性能 |
|------|---------|---------|
| v2 | Python | 100k 交易 ≈ 2s |
| **v3** | **C++** | **100万 交易 ≈ 2s（10x 提升）** |

Beancount v3 保留了 Python API，但解析核心用 C++ 重写，目标是**消除 100 万交易的性能墙**。

### 2. Beanpost：PostgreSQL 后端实验

**Beanpost** 是一个社区项目，将 Beancount 账本**镜像到 PostgreSQL**，并把大量 Beancount 逻辑实现为 SQL 函数。这本质上是在保留 PTA 文本输入的同时，**用数据库解决超大规模查询问题**。

用户报告的使用方式：

- 用 Beanpost 持续同步文本账本 → PostgreSQL
- 用 **Metabase/Tableau** 连接 PostgreSQL 做 BI 分析
- 有人用 **PostGraphile** 自动生成 GraphQL API，上面建 React 前端

### 3. Tackler：Rust 重新实现（性能怪物）

**Tackler** 是用 Rust 从头实现的 PTA 引擎，展示了纯文本解析的极限性能：

| 硬件 | 100万交易处理时间 | 吞吐量 |
|------|----------------|--------|
| MacBook Pro M3 | **1.41 秒** | **700,000+ txns/s** |
| i5-8250U | ~4 秒 | 250,000+ txns/s |

Tackler 比 Ledger (C++) 快 **1.5-2x**，比 hledger (Haskell) 快 **15-20x**，比 Beancount v2 (Python) 快 **30-40x**，内存占用还只有它们的 **1/2~1/4**。

这说明：**PTA 的性能问题不是"纯文本 vs 数据库"的问题，而是具体实现的工程问题**。用 Rust 的内存安全和零成本抽象，PTA 的解析速度可以逼近数据库索引查询。

## 一、PTA 是否有规范？

**简短回答：没有统一的标准规范，但有三个相互独立、部分兼容的"事实规范"，且社区正在推动标准化。**

### 1. 三种"事实规范"

PTA 生态系统目前呈三足鼎立格局，各自的规范程度差异很大：

| 工具 | 规范状态 | 规范载体 | 严格程度 |
|------|---------|---------|---------|
| **Ledger** | ❌ 无正式规范 | **C++ 源码即规范** | 最灵活，验证最少 |
| **hledger** | ✅ 详细文档规范 | [官方手册](https://hledger.org/hledger.html)（~5000 行 manpage） | 中等，兼容 Ledger 大部分语法 |
| **Beancount** | ✅ 最正式的语言规范 | [Martin Blais 的语法文档](https://beancount.github.io/docs/beancount_language_syntax.html) | 最严格，显式声明一切 |

### 2. 各自的核心语法对比

以记录一笔相同的交易（超市购物 100 元）为例：

**Ledger 语法（最自由）：**

```ledger
2024-01-15 超市
    Expenses:Food:Groceries    100.00 CNY
    Assets:Bank:Checking
```

**hledger 语法（与 Ledger 高度兼容）：**

```hledger
2024-01-15 超市
    expenses:food:groceries    100.00 CNY
    assets:bank:checking
```

（hledger 不区分大小写，账户名可小写）

**Beancount 语法（最严格）：**

```beancount
2024-01-15 * "超市" ""
    Expenses:Food:Groceries    100.00 CNY
    Assets:Bank:Checking
```

（Beancount 要求：账户必须先 `open`，交易必须标注 `*` 状态，字符串必须引号包裹）

### 3. 正在进行的 pta-standards 项目

社区已经意识到"无规范"的问题，启动了 **pta-standards** 项目：

🔗 <https://github.com/rustledger/pta-standards>

```text
pta-standards/
├── core/           # 共享基础（数据模型、类型系统、数值精度）
├── formats/        # 各格式规范
│   ├── beancount/  # Beancount v2, v3
│   ├── ledger/     # Ledger v1
│   └── hledger/    # hledger v1
```

这个项目的目标是：

- 提取三种格式的**公共子集**作为"可移植 PTA 核心"
- 定义正式的抽象语法树（AST）规范
- 提供一致性测试套件，验证实现兼容性

### 4. 已有的正式语法定义

## 二、是否能等价映射到数据库操作？

**简短回答：可以，且映射是双向无损的。PTA 的核心数据模型本质上是一个"超图"（hypergraph），可以完美归约为经典的三表关系模型。**

### 1. PTA 的核心数据模型

所有 PTA 工具共享同一套底层数据模型：

```
┌─────────────────────────────────────────────────────────────┐
│                     Transaction（交易）                        │
│  日期 + 描述 + 标记 + [元数据]                                │
│                                                             │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│  │   Posting 1  │  │   Posting 2  │  │   Posting N  │      │
│  │  账户: 金额   │  │  账户: 金额   │  │  账户: 金额   │      │
│  │  [成本] [价格]│  │  [成本] [价格]│  │  [成本] [价格]│      │
│  └──────────────┘  └──────────────┘  └──────────────┘      │
│                                                             │
│  约束：所有 Posting 的金额之和 = 0（复式记账恒等式）            │
└─────────────────────────────────────────────────────────────┘
```

这就是 Kleppmann 所说的图模型：

- **节点（Node）** = Account（账户）
- **超边（Hyperedge）** = Transaction（交易，连接 N ≥ 2 个账户）
- **边标签（Edge Label）** = Posting 的金额（可正可负）

### 2. 到关系型数据库的等价映射

这个超图模型可以无损归约为经典的三表关系模型：

```sql
-- 账户表（节点）
CREATE TABLE accounts (
    id          SERIAL PRIMARY KEY,
    name        TEXT UNIQUE NOT NULL,        -- "Assets:Bank:Checking"
    type        TEXT CHECK (type IN ('Assets', 'Liabilities', 'Equity', 'Income', 'Expenses')),
    currency    TEXT,                         -- 可选的约束货币
    opened_at   DATE,
    closed_at   DATE,
    meta        JSONB
);

-- 交易表（超边）
CREATE TABLE transactions (
    id          SERIAL PRIMARY KEY,
    date        DATE NOT NULL,
    flag        TEXT CHECK (flag IN ('*', '!', 'P')),  -- Beancount 风格
    payee       TEXT,
    narration   TEXT,
    tags        TEXT[],
    links       TEXT[],
    meta        JSONB
);

-- 记账分录表（超边的端点 = 图的边）
CREATE TABLE postings (
    id              SERIAL PRIMARY KEY,
    transaction_id  INT REFERENCES transactions(id),
    account_id      INT REFERENCES accounts(id),
    amount_number   DECIMAL(38, 18) NOT NULL,
    amount_currency TEXT NOT NULL,
    cost_number     DECIMAL(38, 18),          -- 可选的成本基础
    cost_currency   TEXT,
    price_number    DECIMAL(38, 18),          -- 可选的单价
    price_currency  TEXT,
    flag            TEXT,                      -- 单个 posting 的标记
    meta            JSONB,

    -- 复式记账恒等式的数据库级保证
    -- 每笔交易的 postings 金额之和必须为 0
    CONSTRAINT balanced_transaction
        CHECK ((SELECT SUM(p.amount_number)
                FROM postings p
                WHERE p.transaction_id = transaction_id) = 0)
);

-- 价格历史表
CREATE TABLE prices (
    id          SERIAL PRIMARY KEY,
    date        DATE NOT NULL,
    commodity   TEXT NOT NULL,                -- "AAPL"
    currency    TEXT NOT NULL,                -- "USD"
    price       DECIMAL(38, 18) NOT NULL,
    UNIQUE(date, commodity, currency)
);
```

### 3. 实际已有工具做这个映射

#### hledger → SQL（原生支持）

```bash
# 直接导出为 SQL INSERT 语句
hledger print -O sql > ledger_dump.sql

# 直接导入 SQLite
hledger print -O sql | \
    sed 's/id serial/id INTEGER PRIMARY KEY AUTOINCREMENT NOT NULL/g' | \
    sqlite3 ledger.db
```

hledger 生成的 SQL 包含完整的 `transactions` 和 `postings` 表结构，可直接用于 Metabase/Tableau 做 BI 分析。

#### Beancount → SQL（官方工具）

```bash
# Beancount 提供内置工具导出到 SQL
bean-sql my_ledger.beancount sqlite:///path/to/db.sqlite3
```

Beancount 的文档明确说：
> *"Beancount provides a tool to export its contents to a regular SQL database system. Feel free to experiment with it if you like."*

#### Ledger → 自定义脚本

Ledger 没有原生 SQL 导出，但社区提供了 Python/Perl 脚本解析 `.ledger` 文件并导入数据库。

### 4. 映射的等价性证明

**正向映射（PTA → DB）：无损**

- 文本文件中的每一条 `open` → `accounts` 表 INSERT
- 每一条 `YYYY-MM-DD * "..."` → `transactions` 表 INSERT
- 每一条缩进的 `Account  Amount` → `postings` 表 INSERT
- 所有约束（账户必须先 open、交易必须平衡）可以通过数据库 TRIGGER/CHECK 实现

**反向映射（DB → PTA）：无损**

- `accounts` 表 → 按日期排序生成 `YYYY-MM-DD open` 指令
- `transactions` + `postings` 表 JOIN → 按日期排序生成交易块
- 元数据（JSONB）→ 行内 `key: "value"` 语法

**关键洞察**：PTA 文本格式本质上是这个关系模型的**一种特定序列化格式**（human-readable serialization），就像 JSON 是 JavaScript 对象的序列化格式一样。两者之间不存在语义损失，只有表达形式的差异。

### 5. 映射后的能力对比

| 操作 | PTA 文本方式 | 数据库方式 | 优劣 |
|------|------------|-----------|------|
| **余额查询** | 全量解析 → 内存聚合 | `SELECT account_id, SUM(amount) FROM postings GROUP BY account_id` | DB 更快（有索引），但 PTA 在万级数据下无感知 |
| **范围查询** | 全量解析 → 过滤 | `WHERE date BETWEEN '2024-01-01' AND '2024-12-31'` | DB 完胜（B-tree 索引） |
| **多条件筛选** | 命令行参数 → 内存过滤 | SQL WHERE + JOIN | DB 更灵活 |
| **聚合分析** | 有限（内置报表） | 任意 GROUP BY、窗口函数 | DB 完胜 |
| **版本控制** | **Git diff 完美支持** | 数据库迁移工具 | PTA 完胜 |
| **人工可读/编辑** | **直接打开文本文件** | 需工具 | PTA 完胜 |
| **并发写入** | 文件锁（粗粒度） | 事务隔离（ACID） | DB 完胜 |
| **审计追踪** | **Git 历史天然审计** | 需额外审计表 | PTA 完胜 |

### 6. 实践中常见的混合架构

正是认识到两者各有优势，很多重度 PTA 用户采用了**混合架构**：

```
┌─────────────────┐     Git 版本控制     ┌─────────────────┐
│   .beancount    │ ◄──────────────────► │   开发者/会计师  │
│   文本文件       │    （审计、协作）      │   编辑录入      │
│  （真相源）      │                      └─────────────────┘
└────────┬────────┘
         │ 定时同步（cron/Git hook）
         ▼
┌─────────────────┐     SQL/GraphQL      ┌─────────────────┐
│  PostgreSQL     │ ◄──────────────────► │   Metabase/     │
│  （镜像数据库）   │    （分析、BI）       │   Grafana/      │
│                 │                      │   自定义报表      │
└─────────────────┘                      └─────────────────┘
```

**典型案例**：

- 日常记账 → 编辑 `.beancount` 文件（Git 提交）
- 月末分析 → `bean-sql` 同步到 PostgreSQL → Metabase 可视化仪表盘
- 税务申报 → 直接从文本文件生成标准报表（Beancount 原生支持美国 IRS 税表）

## 三、总结

| 问题 | 答案 |
|------|------|
| PTA 有统一规范吗？ | **目前还没有**，但 Beancount 有最正式的语言规范，社区 pta-standards 项目正在推进标准化 |
| 三种格式兼容吗？ | **部分兼容**。hledger 兼容大部分 Ledger；Beancount 语法不兼容但概念完全互通，有 `ledger2beancount` 转换工具 |
| 能映射到数据库吗？ | **完全可以，且是无损双向映射**。核心模型是 Transaction-Posting-Account 三表结构，就是标准的关系型会计数据库设计 |
| 实际有人这么做吗？ | **是的**。hledger 原生支持 `print -O sql`，Beancount 有 `bean-sql` 工具，很多用户采用"文本为源、数据库为分析"的混合架构 |
| 映射后谁更强？ | **互有胜负**：PTA 在版本控制、人工编辑、审计追踪上不可替代；数据库在复杂查询、并发、大规模分析上更强 |

**最终结论**：PTA 不是数据库的"低级替代品"，而是**同一数据模型的两种等价表达**——一种优化了人类可读性和版本控制，一种优化了机器查询和并发处理。最成熟的实践是**以 PTA 文本为"真相源"（source of truth），按需同步到数据库进行 BI 分析**，而不是非此即彼。
