## Context

当前前端交易数据加载以"月"为单位，每次加载整月数据（limit=10000）。问题：
1. 初始加载慢（拉取整月数据）
2. 只能看到最近 1-2 周数据（后端默认 limit=100）
3. 无法浏览更早的历史交易

现有 API `/api/transactions` 已支持 `from`、`to`、`limit` 参数，无需后端改动。

## Goals / Non-Goals

**Goals:**
- 初始加载最新 100 笔交易，记录时间范围
- 滚动到范围边缘时自动触发加载更早数据
- 同一天交易过多时自动膨胀 limit 直到跨天
- 日历点击任意日期可加载该天全部交易
- 重构 store 数据结构支持连续时间范围

**Non-Goals:**
- 日历圆点显示逻辑（保持现状，后续单独优化）
- 分页 UI 组件（用户无感知，自动触发）
- 离线缓存或数据持久化

## Decisions

### 1. 加载粒度：按日期范围 vs 按月

**决策**：按日期范围加载，不以月为单位

**理由**：
- 更精确控制加载量（100 笔而非整月）
- 初始加载更快
- 自然支持"从最新往回拉"的语义
- 避免空月问题（某月无交易时仍需加载）

**替代方案**：保持按月加载
- 优点：与现有 store 结构兼容
- 缺点：初始加载可能跨月，需要多次请求；空月处理复杂

### 2. 滚动触发机制：IntersectionObserver vs 滚动位置检测

**决策**：使用 IntersectionObserver 监听 sentinel 元素

**理由**：
- 更精确（只在 from 日期那组交易进入视口时触发）
- 性能更好（浏览器原生 API，无需计算滚动位置）
- 避免重复触发（sentinel 只触发一次）

**替代方案**：监听 scroll 事件 + 计算距离底部距离
- 优点：实现简单
- 缺点：需要节流/防抖；边界判断不精确

### 3. 同一天膨胀策略：倍增 vs 固定增量

**决策**：limit 倍增（100→200→400→800...）

**理由**：
- 快速收敛（对数级增长）
- 避免过多请求（固定增量可能需要几十次请求）
- 极端情况（某天 10000 笔）只需 7 次请求

**风险**：单次请求数据量可能很大（800+ 笔）
**缓解**：后端 SQLite 查询性能足够；前端渲染虚拟化（未来优化）

### 4. 日历加载策略：独立存储 vs 合并到主列表

**决策**：日历加载的天单独存储在 `calendarDays` Map 中

**理由**：
- 避免污染主列表的时间范围
- 点击过的天可以缓存，无需重复请求
- 展示时合并去重即可

**替代方案**：所有加载都合并到主列表
- 优点：数据结构简单
- 缺点：时间范围不连续，难以追踪；可能加载大量不需要的数据

### 5. 数据结构重构

**当前**：
```typescript
loadedMonths: Set<string>
transactionsByMonth: Map<string, TransactionDto[]>
```

**改造后**：
```typescript
loadedRange: { from: string, to: string } | null
transactions: TransactionDto[]  // 主列表，按时间倒序
calendarDays: Map<string, TransactionDto[]>  // 日历单独加载的天
```

**理由**：
- `loadedRange` 明确记录已加载的时间边界
- `transactions` 是连续数组，便于渲染和 IntersectionObserver
- `calendarDays` 独立存储，不干扰主列表

## Risks / Trade-offs

### [Risk] 同一天膨胀导致单次请求过大
→ **Mitigation**: 倍增策略最多 7 次请求达到 12800 笔；后端 SQLite 查询性能足够；前端渲染性能问题后续用虚拟化解决

### [Risk] 滚动触发时用户可能看到空白区域
→ **Mitigation**: sentinel 放在 from 日期那组交易的末尾，用户看到最后一笔交易时即触发加载；加载期间显示 loading 指示器

### [Risk] 日历加载的天与主列表重复
→ **Mitigation**: 合并展示时按交易 ID 去重；优先使用主列表数据（已加载范围内）

### [Risk] 初始加载时今天无交易
→ **Mitigation**: API 的 `to` 参数语义是"不超过该日期"，会自动返回之前最新的交易，符合预期

### [Trade-off] 不修改日历圆点逻辑
→ 圆点仍只显示主列表范围内的日期，日历加载的天不会显示圆点（除非该天在主列表范围内）
→ 后续可单独优化圆点逻辑
