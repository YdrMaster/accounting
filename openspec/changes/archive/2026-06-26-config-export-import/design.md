## Context

当前项目使用 SQLite 作为本地数据库，通过 `accounting-sql` crate 管理表结构，`accounting-service` 封装业务逻辑，`accounting-cli` 提供命令行入口。数据库中既有账本数据（交易、分录、附件等），也有大量配置数据（账户树、币种、成员、渠道、标签、预算、导入映射等）。

用户需要一种机制，将配置数据导出为单一、人类可读可写的 YAML 文件，并能在不同数据库之间迁移和合并这些配置，而不会影响账本流水。

## Goals / Non-Goals

**Goals:**
- 提供 `config export <file>` 和 `config import <file>` CLI 命令。
- 导出除账本外的所有配置到单个 YAML 文件。
- 支持按自然键合并导入，不删除目标库已有配置。
- 保证导入的原子性（全成功或全回滚）。
- 使用 YAML 作为导出格式，金额使用字符串避免浮点误差。

**Non-Goals:**
- 不导出/导入交易、分录、附件、交易标签、渠道路径。
- 不处理 API 或 Web UI 的导入导出入口。
- 不做跨币种换算或预算实际值计算。
- 不实现配置差异对比或版本迁移升级逻辑（仅保留 `version` 字段占位）。

## Decisions

### 1. 使用 YAML 而非 JSON/TOML
**选择**: YAML。

**理由**:
- 可读性和可写性最佳，接近自然文本。
- 支持注释，方便用户手写说明。
- 对嵌套列表和账户树表达友好。TOML 的数组表在此场景下过于啰嗦；JSON 不支持注释且引号/括号较多。

**替代方案**: JSON（已有依赖，但可读可写性弱）；TOML（支持注释，但嵌套结构冗长）。

### 2. 在 service 层定义 YAML DTO，不污染 core 类型
**选择**: 在 `accounting-service` 中新建 `config` 模块，定义 `ConfigExport`、`ConfigImport` 等 DTO 结构体，并为其派生 `Serialize`/`Deserialize`。

**理由**:
- `accounting` core crate 目前不依赖 serde，保持其独立和轻量。
- YAML 的字段命名和结构（如账户路径字符串）与内部树形 ID 模型不同，DTO 可以清晰隔离外部协议与内部模型。

**替代方案**: 直接给 `accounting` 核心类型加 serde derive。该方案会扩大 core crate 依赖，且核心类型使用 `AccountId` 等内部 ID，无法直接映射到自然键。

### 3. 导出时忽略 is_system，依赖目标库已初始化
**选择**: 导出文件不包含 `is_system` 字段；导入时不修改系统内置记录的 `is_system` 状态。

**理由**:
- 目标数据库已经完成初始化，内置数据已存在。
- 系统内置账户/标签/渠道在不同语言种子数据中名称不同，导出 `is_system` 会在跨语言导入时产生混乱。
- 简化用户手动编辑配置文件的认知负担。

### 4. 账户路径作为自然键，路径变更视为新账户
**选择**: 账户以完整路径 `Assets:Bank:Checking` 表示。导入时按路径查找；路径不存在则创建（包括自动创建缺失的父账户）。

**理由**:
- 路径是人类理解和编辑账户树最直接的方式。
- 若将路径变更解释为“重命名”，会隐式影响账本中的分录引用，与“不改动账本”的目标冲突。

### 5. 导入使用单事务
**选择**: 整个导入流程在 `sqlx::Transaction` 中执行，失败统一回滚。

**理由**:
- 配置表之间存在外键依赖，部分成功会导致数据库处于不一致状态。
- 事务保证配置迁移的可靠性。

### 6. 金额使用字符串序列化
**选择**: YAML 中的 `budget_limits` 金额使用 `"3000.00"` 字符串。

**理由**:
- 避免 YAML 数字解析为 f64 导致的精度损失。
- `rust_decimal::Decimal` 可通过字符串精确构造。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| YAML 缩进语法对用户不友好，容易写错 | CLI 导入失败时返回清晰的行号和字段错误信息；未来可考虑提供 JSON schema 或校验命令 |
| 跨语言数据库导入时，系统账户路径可能不匹配 | 导入前强制检查 `settings.language`，语言不一致即拒绝；系统内置数据不在配置文件中管理 |
| 账户路径冲突（如大小写、空格） | 保留现有数据库对 `accounts.name` 的唯一约束，导入时按路径精确匹配 |
| 大配置文件导入时内存占用高 | 当前配置数据量有限（账户/标签/渠道数量级小），单文件 YAML 可接受；未来若数据量剧增再考虑流式处理 |
| 导入失败回滚后用户不知道哪些数据有问题 | 在事务提交前先做引用完整性校验，失败时返回具体失败的实体和路径 |

## Migration Plan

本变更不修改现有表结构，仅新增 CLI 命令、Service 模块和 repo 查询方法。部署步骤：

1. 在 workspace `Cargo.toml` 中添加 YAML 序列化依赖。
2. 在 `accounting-service` 中实现 `ConfigService`。
3. 在 `accounting-sql` 中补齐配置相关 repo 方法。
4. 在 `accounting-cli` 中新增 `config` 子命令。
5. 运行现有测试，确保不破坏已有功能。

回滚策略：未改动数据库 schema，回滚只需回退代码即可。
