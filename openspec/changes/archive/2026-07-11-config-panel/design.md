## 背景

记账 Web 应用使用窗格式布局（ResponsiveShell），顶部 PageSwitcher 负责导航切换。现有的覆盖层模式包括：
- **底部抽屉**（AccountDrawer）：`position: absolute; inset: 0`，内容从底部滑入，`max-height: 50vh`，顶部圆角
- **全屏覆盖层**（TransactionFormOverlay、AccountPickerOverlay）：覆盖整个视口

基础数据（成员、渠道、标签）目前在前端只有只读的 store，后端 CRUD 不完整：成员缺少改名接口；标签缺少更新接口；渠道只能更新 account_id。

## 目标 / 非目标

**目标：**
- 提供统一入口（PageSwitcher 中的齿轮图标）管理基础数据
- 复用底部抽屉模式实现配置面板 UI
- 支持成员和标签的行内 CRUD（简单的基于名称的实体）
- 支持渠道的可展开卡片 UI（多字段实体，含账户关联）
- 补全后端 rename/update 接口缺口

**非目标：**
- 批量操作（导入/导出）
- 删除的审计历史或撤销功能
- 拖拽排序
- 将配置面板做成常驻窗格（用户明确要求弹出式）

## 决策

### 1. 底部抽屉 + 66vh 最大高度（而非全屏覆盖或侧边抽屉）

**决策**：复用现有底部抽屉模式（类似 AccountDrawer），但将 max-height 增大到 66vh。

**理由**：底部抽屉已在应用中建立。66vh 为列表内容提供足够空间，同时保持页面上下文可见。全屏覆盖对轻量配置来说太重；侧边抽屉与水平窗格滑动交互冲突。

**考虑过的替代方案**：
- 全屏覆盖层：对轻量配置操作来说过于沉重
- 右侧抽屉：与水平窗格滑动交互冲突
- 独立窗格：用户明确要求非常驻、弹出式

### 2. 抽屉内 tab 导航（而非子视图或手风琴）

**决策**：抽屉 body 顶部放三个水平 tab（成员/渠道/标签）。

**理由**：只有三个类别，且互相对等。tab 是这种情况下最简单的导航方式。手风琴浪费垂直空间；子视图增加返回导航的开销。

### 3. 成员/标签行内编辑，渠道可展开卡片

**决策**：成员和标签使用行内改名（点击名称 → 输入框 → 回车保存）。渠道使用可展开卡片，展开后显示所有字段。

**理由**：成员和标签基本只有名称（标签有可选描述）——行内编辑快速自然。渠道有三个字段（name、description、account_id）需要更多空间；可展开卡片保持列表紧凑，同时允许详细编辑。

### 4. 关联账户通过复用 AccountPickerOverlay 实现

**决策**：渠道的"关联账户"字段使用"选择"按钮打开现有的 AccountPickerOverlay。

**理由**：AccountPickerOverlay 已处理账户浏览、类型分组和选择。复用它避免构建新组件，确保一致性。

### 5. 后端 API：扩展现有端点而非新建

**决策**：
- 成员：新增 `PUT /api/members/{id}`，body 为 `{ name }`
- 标签：新增 `PUT /api/tags/{id}`，body 为 `{ name, description }`
- 渠道：扩展现有 `PUT /api/channels/{id}`，接受 `{ name, description, account_id }`（目前只接受 `account_id`）

**理由**：遵循 RESTful 约定，最小化新路由，向后兼容（现有字段仍然被接受）。

### 6. 删除操作不做确认弹窗

**决策**：删除操作立即执行，不弹确认框。

**理由**：这些都是简单的基础数据项。后端已有引用完整性检查（如被交易引用的渠道不能删除）。为每次删除添加确认弹窗增加摩擦但安全收益不大。

## 风险 / 权衡

- **[渠道展开/折叠状态管理]** → 保持简单：用单个 `expandedId` ref 追踪，同时只展开一张卡片。如果体验受限，后续可放宽。
- **[后端改名可能影响前端缓存状态]** → 任何变更后强制重新加载相关 store（`store.load(true)`）。现有 store 已支持此模式。
- **[抽屉 z-index 与其他覆盖层冲突]** → 配置抽屉和 AccountPickerOverlay 不应同时打开。处理方式：从配置抽屉内打开 AccountPickerOverlay 时，AccountPicker 在抽屉的 z-index 上下文中渲染。
