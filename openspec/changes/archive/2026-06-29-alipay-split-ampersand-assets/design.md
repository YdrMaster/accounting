## Context

`accounting-service/src/import/alipay.rs` 中的 `parse_alipay_row` 目前把 CSV 第 7 列 `收/付款方式` 的完整字符串作为单一 Asset Posting 的 `category`。当该字段包含 `&` 时（如 `蚂蚁宝藏信用卡(江苏银行)&超划算`），整个字符串被当作一个账户名，映射 key 为 `资产:蚂蚁宝藏信用卡(江苏银行)&超划算`，用户无法为其中的信用卡、优惠券/补贴分别设置映射。

下游 `ImportService::submit_entry` 已经支持任意数量的 `BillPosting`，因此拆分只影响适配器层，不需要改动服务编排逻辑。

## Goals / Non-Goals

**Goals:**
- 让 `&` 分隔的支付宝付款方式生成多个独立的 Asset Posting；
- 每个部分都能独立命中账户映射或生成 `Import:支付宝:资产:<part>` 子账户；
- 保持交易金额平衡，主资产Posting 的金额与实际扣款一致；
- 为空字段保留原有回退行为（使用渠道名）。

**Non-Goals:**
- 不推断 `&` 各部分的实际金额（CSV 未提供该信息）；
- 不改写 `BillAdapter` trait、`ImportService`、`MappingService`、数据库 schema；
- 不处理历史已导入数据；
- 不处理 `&` 在 `交易分类`（收支侧）出现的情况（实测未出现）。

## Decisions

### 1. 拆分位置：支付宝适配器内部

**选择**：在 `AlipayAdapter::parse_alipay_row` 中拆分 `收/付款方式`。

**理由**：
- `&` 是支付宝 CSV 的具体格式约定，属于适配器职责；
- `ImportService` 和 `BillAdapter` trait 保持通用，不需要为单一适配器引入特殊逻辑。

**替代方案**：在 `ImportService` 中根据映射规则拆分。
- 放弃原因：会把 Alipay 的格式细节泄漏到通用服务层，且映射层无法表达“一个分类对应多个账户”的语义。

### 2. 金额分配：全额给第一部分，其余为 0

**选择**：第一个非空部分承担全部 `asset_amount`，其余部分生成金额为 `Decimal::ZERO` 的 Asset Posting。

**理由**：
- 真实资金只从第一个账户流出（或流入，退款场景），与 CSV 中的总金额一致；
- 0 金额 Posting 在当前 `validate_transaction` 规则下被允许（普通分录不检查非零，仅退款冲减分录检查非零）；
- 其余部分（优惠券、立减金等）虽然金额为 0，但能独立生成账户并参与映射，方便用户后续手动调整。

**替代方案**：
- **平均分配金额**：放弃。实测 `&` 的一边往往是优惠券/补贴，平均分配会扭曲真实资金流向。
- **仅保留第一部分，丢弃其余**：放弃。用户明确要求“按 `&` 分开”，丢弃其余无法分别映射。
- **拆分为多笔交易**：放弃。会破坏原始交易的一一对应关系，且重复 expense 不合法。

### 3. 空字段回退行为不变

**选择**：当 `收/付款方式` 为空或拆分后全部为空时，仍使用 `channel_name` 作为单一 Asset Posting 的 `category`。

**理由**：保持与现有行为一致，避免引入新的空分类账户。

## Risks / Trade-offs

- **[0 金额 Posting 的语义] →  mitigation**：0 金额Posting 在系统中合法，但可能在报表或 UI 中显示为“参与交易但无金额”。由于导入交易默认带 `待处理` Tag，用户会在确认阶段发现并调整。
- **[用户已有复合名称映射失效] →  mitigation**：原映射 `资产:蚂蚁宝藏信用卡(江苏银行)&超划算` 不再被命中。这是预期行为；用户需要为拆分后的单独部分重新设置映射。可在变更说明中强调。
- **[多个 0 金额账户污染账户树] →  mitigation**：拆分出的部分若无映射会创建 `Import:支付宝:资产:<part>` 子账户。由于这些交易带 `待处理` Tag，用户确认后通常会清理或重新映射。
- **[未来 `&` 含义变化] →  mitigation**：拆分逻辑集中在 `AlipayAdapter`，若支付宝后续调整格式，只需修改适配器。

## Migration Plan

无需迁移。本变更只影响新导入的支付宝交易，历史数据保持不变。
