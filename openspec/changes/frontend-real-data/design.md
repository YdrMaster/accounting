## Context

前端为 Vue 3 + TypeScript + Pinia，无 vue-router（自定义 ResponsiveShell 横向排列 5 个 pane）。后端为 Rust Axum + SQLite，所有 API 端点已就绪。当前 3 个页面（资产、日历、预算）使用硬编码数据，交易页只读，账户页缺少新建功能。

现有 UI 模式：AccountDrawer 底部滑出抽屉（max-height 50vh, max-width 600px），手写 CSS 暗色主题，原生 fetch API。历史分支（`history`）有完整的 TransactionForm.vue 和 AccountPicker.vue 参考实现，但使用 Ant Design Vue 和 vue-router，需适配当前架构。

## Goals / Non-Goals

**Goals:**
- 所有 5 个页面接入真实后端数据
- 交易支持完整复式记账的创建/编辑/删除
- 交易列表和交易卡片组件在交易页和日历页之间共享
- 账户选择体验复用现有账户网格

**Non-Goals:**
- 退款/报销模式（后续迭代）
- vue-router 集成或深链接
- 多币种 cost 编辑（后端支持但前端暂不暴露）
- 附件管理
- 移动端适配优化（当前仅桌面端）

## Decisions

### Decision 1: 交易表单使用覆盖层面板而非底部抽屉

**选择**: 覆盖层面板（覆盖整个 pane 区域）

**替代方案**: 底部抽屉（与 AccountDrawer 一致）

**理由**: 交易表单包含日期、备注、成员、标签、渠道链路、多条分录，内容远超账户编辑的复杂度。50vh 的抽屉高度不足以舒适地编辑多分录交易。覆盖层面板在 pane 内提供完整编辑空间，同时不干扰其他 pane。

### Decision 2: 账户选择使用覆盖层复用账户网格

**选择**: 点击 AccountPicker 触发器 → 打开覆盖层 → 显示现有账户网格（AccountGrid）→ 选中高亮 + 确认按钮

**替代方案 A**: 下拉树形选择器
**替代方案 B**: 历史分支的 BottomSheet + AccountTreeList

**理由**: 账户网格是用户已熟悉的交互模式（账户页的核心组件），复用它可以保持体验一致性，且不需要新建树形选择组件。覆盖层方式避免在 pane 内嵌套过深的弹出层。

### Decision 3: 交易列表抽取为共享组件

**选择**: 提取 TransactionList + TransactionCard 为独立组件，TransactionView 和 CalendarView 分别嵌入

**理由**: 两个页面的交易展示逻辑完全相同（日期分组、卡片样式、展开/折叠、增删改交互），仅数据来源和导航方式不同。共享组件避免重复代码，确保两个页面的交易体验一致。

### Decision 4: 交易页使用连续滚动而非按月分页

**选择**: 连续滚动，滚动到边界自动加载相邻月份数据

**替代方案**: 按月分页滑动（一次显示一个月）

**理由**: 连续滚动提供更自然的浏览体验，用户不需要显式切换月份。实现上通过 IntersectionObserver 或 scroll 事件检测边界，触发 API 加载。

### Decision 5: channel_path 输入采用分级标签模式

**选择**: 每一级是一个槽位，内含渠道标签 chips。前几级单选，最后一级多选。

**理由**: 后端模型支持同一 position 多个 channel_id（最后一级的 `&` 语法），前端用标签 chips 自然映射这个结构。添加新一级时，原最后一级变为单选。

### Decision 6: 不平衡时分录提交按钮禁用

**选择**: 不显示合计行，仅通过禁用确认按钮阻止不平衡提交

**理由**: 复式记账用户天然理解借贷平衡。额外的合计显示增加视觉噪音。按钮禁用已足够传达"不能提交"的信号。

### Decision 7: 覆盖层定位规则 — 所有覆盖层都只在 pane 内，不覆盖整个屏幕

**选择**: 所有覆盖层（交易表单、账户选择器）都使用 `position: absolute` 相对于父容器定位，不使用 `position: fixed`

**规则**:
1. **交易表单覆盖层** → `position: absolute`，覆盖交易 pane（父容器 `.transaction`）
2. **账户选择器覆盖层** → `position: absolute`，覆盖交易表单（相对于交易 pane，不是整个视口）
3. **没有任何覆盖层应该占据整个屏幕** → 所有覆盖层都只在各自的 pane 内，不影响其他 pane
4. **滚动区域** → 当表单显示时，交易 pane 的滚动区域变成表单本身（父容器 `overflow: hidden`）

**层级关系**:

**禁止**:
- ❌ 使用 `position: fixed` 让任何覆盖层占据整个视口
- ❌ 覆盖层影响其他 pane 的显示或交互
- ❌ 滚动时底层内容从覆盖层后面暴露出来

**理由**: 
- 保持每个 pane 的独立性，覆盖层只影响当前操作的 pane
- 用户可以同时看到其他 pane 的内容（如资产、账户）
- 避免全屏覆盖带来的迷失感和交互混乱

## Risks / Trade-offs

- **[连续滚动性能]** 大量交易可能导致 DOM 节点过多 → 使用虚拟滚动或限制加载范围（每次加载 1 个月），卸载远离视口的月份 DOM
- **[覆盖层嵌套]** 交易表单覆盖层内再打开账户选择覆盖层，两层叠加可能产生交互混乱 → 账户选择覆盖层 z-index 更高，关闭时恢复表单状态
- **[交易表单复杂度]** 完整复式记账表单涉及分录增删、自动配平、账户选择、标签、渠道，开发量大 → 先实现 normal 模式核心流程，退款/报销后续迭代
- **[历史分支代码复用]** 历史分支使用 Ant Design Vue，当前主线手写 CSS，无法直接复用 → 仅参考业务逻辑（自动配平、提交验证），UI 层重写
